## Canonical Business Context (`01-Areas/_context/`)

Load these before ANY content, copy, or marketing work. They are the single source of truth.

### Brand Voice (`brand-voice.md`)

**Personality:** Direct, confident, slightly provocative, conversational, educational, premium — all mixed. Sound like a smart founder talking to another smart founder.

**Tone:** Polished enough to be taken seriously. Raw enough to feel real. Like a well-edited voice note — not a PR statement, not a Twitter rant.

**Use:** Short punchy sentences. Dry humour (occasional). Conversational but not sloppy.

**Words to use:** "honest", "specific", "actionable", "real", "actual", "without the [noise/fluff]", short imperatives.

**Words to avoid:** "leverage", "utilize", "synergy", "game-changer", "revolutionary", "we believe", trailing qualifiers.

**Good example:**
> "Submit your creator profile. Get roasted. Fix what's broken."

**Bad example:**
> "We're leveraging AI to revolutionize how creators get feedback on their journey!"

---

### ICP (`icp.md`)

**Who:** Indie hacker, solo founder, builder-marketer hybrid. Medium-to-high technical level. Post-MVP, has something real, doesn't know what to do with it.

**Goal:** Ship something that gets traction — paying users, audience, proof it's not a waste of time.

**Top pain points:**
1. Can't market their product — good at building, bad at positioning
2. Wasting time on manual tasks that should be automated
3. No honest signal on quality — friends say "looks great", strangers say nothing
4. Overwhelmed, no system — context dies between sessions

**How they talk:** "I just need someone to tell me what's actually wrong." / "I keep starting over every session."

**Where to find them:** X (building in public), Indie Hackers, Hacker News, Product Hunt, Discord (Buildspace, MakerPad).

---

### Positioning (`positioning.md`)

**One-liner:** Tools that give indie founders honest, actionable clarity — no fluff, no polite noise, no wasted time.

**Category:** Honest feedback + automation tools for solo founders. At the intersection of "things that tell you the truth" and "things that run without you."

**Core differentiator:** Honesty + specificity. The market tells founders what they want to hear. The positioning: tell them what's actually wrong, with enough specificity to act on immediately.

## Knowledge Capture Flow

Fully automatic — zero manual effort required.

```
Session ends
     │
     ▼
SessionEnd hook → flush.py
     │  Extracts concepts + connections from session transcript
     ▼
daily/YYYY-MM-DD.md  (raw daily log)
     │
     ▼  (after 6 PM — compile.py runs)
knowledge/concepts/  +  knowledge/connections/
     │
     ▼  (symlinks)
03-Resources/Concepts/  ←→  Obsidian
03-Resources/Connections/ ←→ Obsidian
05-Daily-Logs/           ←→ Obsidian (symlink to daily/)
     │
     ▼
auto-session-save.sh
     Writes session handoff → 02-Projects/Claude Sessions/YYYY-MM-DD-[slug].md
     │
     ▼  (next session)
SessionStart hook
     Injects knowledge index into context automatically
```

## Note Conventions

**Wikilinks:** `[[note-slug]]` — filename without `.md` extension. Used for all internal links.

**YAML frontmatter:** Required on every note. Standard fields:
```yaml
---
title: Note Title
type: context | area | project | spec | concept
status: active | paused | archived
updated: YYYY-MM-DD
tags: [tag1, tag2]
---
```

**Project @-imports in CLAUDE.md** (auto-load context at session start):
```markdown
@~/MyVault/01-Areas/_context/brand-voice.md
@~/MyVault/01-Areas/_context/icp.md
@~/MyVault/02-Projects/[project]/index.md
```

## Session Rituals

### `/resume`
Reads `CLAUDE.md` + latest session handoff + active project indexes. Returns precise current status — what's done, what's next, what to open.

### `/wrap-up`
- Summarizes the session
- Updates `CLAUDE.md` → "Current State & Handoffs" section
- Creates handoff note in `02-Projects/Claude Sessions/`
- Links new files created during session

## Tool Routing — Critical Rule

| Task | Use | Never Use |
|---|---|---|
| Search vault by keyword | `ctx_search(pattern, "~/MyVault")` | `mcp__obsidian__` |
| Read a vault note | `ctx_read("~/MyVault/path/to/note.md")` | `mcp__obsidian__` |
| Map vault structure | `ctx_tree("~/MyVault", depth)` | `ls` / `find` |
| List area/project files | `ctx_tree("~/MyVault/01-Areas", 2)` | `mcp__obsidian__` |

**Why:** lean-ctx reads the same markdown files faster, with session caching and token compression. `mcp__obsidian__` is never used — lean-ctx is the mandatory tool for all vault access.

---

## Context Loading Order (Before Any Project Work)

1. Check project `CLAUDE.md` for `@path` imports — if present, context is already loaded
2. If not, `ctx_search` for the project/product name to find relevant notes
3. Load `01-Areas/_context/` canonical docs (brand voice, ICP, positioning, products)
4. Read `02-Projects/[project]/index.md` for current status
5. Never ask the user for information that may already be in the vault — search first
