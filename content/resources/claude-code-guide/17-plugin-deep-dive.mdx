## Agent Alchemy (sequenzia) â€” 6 Sub-Plugins

Agent Alchemy is a suite of six coordinated plugins from `sequenzia` that implement structured development workflows using Claude Code's native Agent Teams, task management (`TaskCreate`/`TaskUpdate`/`TaskList`/`TaskGet`), and `SendMessage` coordination.

**Cross-plugin loading:** Skills routinely load siblings via `Read ${CLAUDE_PLUGIN_ROOT}/../<other-plugin>/skills/<skill>/SKILL.md`. The full suite is designed to work as a unit.

**Settings file:** `.claude/agent-alchemy.local.md` â€” controls per-workflow approval behavior, cache TTLs, TDD strictness, max parallelism, and output paths.

## Hookify Plugin

**Location:** `~/.claude/plugins/cache/claude-plugins-official/hookify/7f3389d21fa9/`

**Purpose:** Create and manage custom behavior hooks via markdown rule files â€” no `hooks.json` editing required.

### How It Works

Hookify registers Python hook scripts for all four hook events (PreToolUse, PostToolUse, Stop, UserPromptSubmit). These scripts scan `.claude/` for `hookify.*.local.md` files and evaluate their rules against each event.

**Rule files:** `.claude/hookify.{rule-name}.local.md`

**Rule format:**
```markdown
---
name: rule-identifier
enabled: true
event: bash|file|stop|prompt|all
pattern: regex-pattern
action: warn|block
---
Message to show when rule triggers.
```

Multi-condition rules use a `conditions:` YAML array instead of `pattern:`, with `field` + `operator` + `pattern` per condition. All conditions must match.

**Events:** `bash` (Bash tool), `file` (Edit/Write/MultiEdit), `stop` (agent stopping), `prompt` (user prompt submit), `all`.

**Actions:** `warn` (show message, allow operation) or `block` (prevent execution / stop session).

**No restart needed:** Rules take effect on the next tool use after file creation.

### Active Rules

Hookify rules are only active when `hookify.*.local.md` files exist â€” in `~/.claude/` for global rules or in a project's `.claude/` directory for project-scoped rules. A fresh install has none; you create them on demand via `/hookify`.

### Commands

| Command | Description |
|---------|-------------|
| `/hookify [behavior]` | Create rules from explicit instructions or (without args) analyze recent conversation for unwanted behaviors via `conversation-analyzer` agent. Interactive: presents findings, asks warn vs block, creates `.claude/hookify.{name}.local.md`. |
| `/hookify:configure` | Enable/disable existing rules interactively. Lists all `.claude/hookify.*.local.md` with current state, lets user toggle with AskUserQuestion. |
| `/hookify:list` | List all configured rules with status. |
| `/hookify:help` | Usage guide. |

### Skill

| Skill | Description |
|-------|-------------|
| `hookify:writing-rules` | Full reference for rule file format, frontmatter fields, event types, operators (`regex_match`, `contains`, `equals`, `not_contains`, `starts_with`, `ends_with`), field reference per event type, and writing tips. Loaded by the hookify command before rule creation. |

### Conversation-Analyzer Agent

The `conversation-analyzer` agent (model: inherit, tools: Read + Grep) scans the current conversation for:
- Explicit requests to avoid something ("don't use X", "stop doing Y")
- Frustrated reactions ("why did you do X?")
- Corrections/reversions (user fixing Claude's actions)
- Repeated issues

Returns structured findings: category, tool involved, regex pattern, context, severity (high/medium/low). The `/hookify` command then filters by severity and creates rule files. The agent deliberately avoids false positives from hypotheticals, teaching moments, or one-time accidents.

## claude-mem Plugin (v10.5.5)

**Location:** `~/.claude/plugins/cache/thedotmack/claude-mem/10.5.5/`

**Purpose:** Persistent cross-session memory database with AST-based code exploration. Stores observations, decisions, bugs, and discoveries as structured data accessible across sessions.

### MCP Tools

The plugin exposes an MCP server with these tools:
- `search(query, limit, project, type, obs_type, dateStart, dateEnd, offset, orderBy)` â€” Returns index with IDs, timestamps, types, titles (~50-100 tokens each). Types: "observations", "sessions", "prompts". `obs_type`: bugfix, feature, decision, discovery, change.
- `timeline(anchor, query, depth_before, depth_after, project)` â€” Chronological view around an anchor observation. Interleaves observations + sessions + prompts.
- `get_observations(ids, orderBy, limit, project)` â€” Batch fetch full observation details (~500-1000 tokens each). Always use for 2+ observations.
- `smart_search(query, path, max_results, file_pattern)` â€” AST-based code search. Returns ranked symbols with signatures and line numbers + folded file views.
- `smart_outline(file_path)` â€” Structural skeleton of a file (all functions, classes, methods, imports).
- `smart_unfold(file_path, symbol_name)` â€” Full source of a single symbol with complete implementation.

### Commands

| Command | Description |
|---------|-------------|
| `/claude-mem:mem-search` | 3-layer search workflow: search â†’ timeline for context â†’ get_observations for selected IDs. 10x token savings vs fetching everything. |
| `/claude-mem:make-plan` | ORCHESTRATOR mode: deploys Documentation Discovery subagents first (Phase 0 always), then phased implementation. Subagents must return sources, concrete API names, copy-ready snippet locations, and confidence notes. Rejects reports without sources. |
| `/claude-mem:do` | ORCHESTRATOR mode: executes a plan via subagents. Implementation â†’ Verification â†’ Anti-pattern check â†’ Code quality â†’ Commit (only after verification passes). Each phase uses fresh subagents. Failure modes prevented: no inventing APIs, no undocumented parameters. |
| `/claude-mem:smart-explore` | Activates the `smart-explore` skill. Overrides default exploration: use `smart_search` â†’ `smart_outline` â†’ `smart_unfold` instead of Read/Grep/Glob. 4-8x token savings on file understanding. Index first, fetch on demand. |

### 35 Language Modes

The `modes/` directory contains 35 specialized response-style configurations:
- **Code modes:** `code.json` (default), `code--chill.json` (relaxed tone), plus 30 language-localized variants: ar, bn, cs, da, de, el, es, fi, fr, he, hi, hu, id, it, ja, ko, nl, no, pl, pt-br, ro, ru, sv, th, tr, uk, ur, vi, zh.
- **Specialty modes:** `email-investigation.json`, `law-study.json`, `law-study--chill.json`.
- These JSON files configure response tone, format, and language for different contexts.

### Session Population

The plugin uses hooks (`hooks.json`) for automatic session context collection. The `session-end-cleanup` hook archives session data. Cross-session memory is populated from:
1. Observations explicitly written during sessions
2. Session transcripts indexed at end-of-session
3. Prompt history tracked via UserPromptSubmit hook
