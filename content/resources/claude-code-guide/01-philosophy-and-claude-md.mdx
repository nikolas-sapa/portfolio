> **New here?** Claude Code is a command-line tool that lets Claude run shell commands, edit files, and build on your machine. Out of the box it's generic. `CLAUDE.md` is the file that teaches Claude how **you** work — your rules, your stack, your preferences. Set it up once and every session starts already knowing what matters.

## 1. Identity & Working Style (from `~/MyVault/CLAUDE.md`)

**Who:** the author — builder, marketer, developer. Runs marketing funnels, builds Claude Code automation (skills, hooks, agents), and ships web products. Thinks in systems; wants everything automated or templated.

**Stack:** Next.js · TypeScript · Tailwind · Vercel · n8n · Telegram · Meta Ads · Obsidian

**Tools:** Claude Code (heavy user) · Superpowers skills · MCP (Notion, GitHub, Gmail, Playwright, Apify)

**Core priorities (in order):**
1. Correctness
2. Maintainability
3. Performance
4. New features

**Working style:**
- Simple over clever. One thing done right beats three things half-done.
- Concise responses — no trailing summaries, no filler.
- Parallelize everything that can be parallelized.

## 2. Global Rules (`~/.claude/CLAUDE.md`)

### Package Manager
- Use `npm` exclusively — **never** yarn, pnpm, or bun.

### Debugging Protocol
- Verify infrastructure first (network, auth, env vars, API access) **before** attempting code fixes.

### Installation Discipline
- When installing skills/plugins/packages, install ONLY what was explicitly requested — no exploring or adding extras.

### Autonomy Boundaries
- **Act independently on:** running commands, file edits, fixing your own errors, obvious decisions.
- **Confirm before:** anything destructive, irreversible, or visible to others — pushes, deploys, deletions, messages, scope changes.

### General Code Rules (also in `~/MyVault/CLAUDE.md`)
- Always check for applicable skills before acting — even 1% chance = invoke.
- Use `context7` before writing code for any library.
- Prefer editing existing files over creating new ones.

## 3. Session Rituals

### Session Start
Output a title at the start of **every** session:
```
Session: [snake_case_title]
```

### `/resume` command
Reads `~/MyVault/CLAUDE.md` + latest handoff note + active project indexes. Gives a precise status report. Trigger: user runs `/resume`.

### `/wrap-up` command
Summarizes session, updates `~/MyVault/CLAUDE.md`'s "Current State" section, creates handoff note, links new files. Trigger: user runs `/wrap-up`.

### Session End Protocol (self-healing)
- **Self-heal:** If any agent/skill underperformed, propose edit in `<rule>` tags.
- **Improve orchestrator:** If delegation went well or poorly, update `~/.claude/agents/orchestrator.md`.

## 4. Plan → Execute → Verify Workflow

Apply for **non-trivial tasks** (>1 file, unclear scope, risky changes):

1. **Plan** — use `Plan` agent or `superpowers:writing-plans`
2. **Execute** — implement per plan; parallelize independent steps
3. **Verify** — run `superpowers:verification-before-completion` before claiming done; verify actual behavior, not a proxy for it

> **Rule: Never claim done without verifying the exact production behavior requested.**

## 5. Skill Gate Rules

> USER INSTRUCTIONS OVERRIDE `using-superpowers`. These rules are law.

### `superpowers:brainstorming`
ONLY fire when **ALL** of these are true:
- User is starting something new (new product, new feature, new app, new system)
- User hasn't defined what to build yet
- Request uses words like: "build", "create", "design", "make", "I want to", "let's make"

**NEVER fire brainstorming for:** audits, analysis, "check X", "fix X", "why does X", "explain X", "optimize X", "what's wrong with X", debugging, code review, refactoring, config changes, Q&A, single-file changes, setup reviews.

### `superpowers:writing-plans`
Fire when:
- User explicitly asks to plan something
- Task spans >3 files AND has unclear implementation path

### `superpowers:systematic-debugging`
Fire when:
- Bug is non-obvious after reading the error
- Problem has been attempted once and failed

### `superpowers:verification-before-completion`
Fire when:
- A multi-file implementation is complete and needs validating

### `frontend-design:frontend-design`
Fire when:
- Building a new UI page or major component from scratch
- User gives visual/design feedback on existing UI

### Other skill triggers (only when explicitly needed)
| Skill | Condition |
|---|---|
| `claude-mem:mem-search` | Prior session context needed |
| `update-config` | Changing settings, hooks, or permissions |
| `resume` | User runs `/resume` |
| `wrap-up` | User runs `/wrap-up` |

## 6. Advisor Gate Rules

> `advisor` uses Opus — expensive. Only call it when the decision genuinely requires deeper reasoning.

### CALL advisor when:
- Cross-system architecture decision (auth strategy, data model design, infra choice)
- Security-sensitive design (auth flows, encryption, API security)
- Stuck after 2+ failed attempts at a bug
- Major refactor spanning >10 files

### NEVER call advisor for:
- Audits, analysis, or setup review tasks
- Single-file changes or config edits
- Routine implementation (components, API routes, migrations)
- Questions answerable by reading code or docs
- Tasks not yet attempted

## 8. Context Budget Rules

- After every response, mentally check session length.
- **Trigger warning if:** 20+ back-and-forth exchanges completed OR user is switching to a clearly unrelated new task.
- Warning text to output:
  > "Context is getting large — run `/compact` to summarize history and reduce cost before we continue."
- Never let context grow silently past the point where it costs more than the work is worth.

### Context Discipline (subagents)
- Compress context passed to subagents — never forward raw output.

## 10. lean-ctx — Context Engineering Layer

> Source: `~/.claude/CLAUDE.md` (injected) + `~/CLAUDE.md` (home project file)

### Mandatory Tool Mapping (CRITICAL — not optional)

| ALWAYS USE | NEVER USE | Why |
|---|---|---|
| `ctx_read(path)` | `Read` / `cat` / `head` / `tail` | Cached, 8 compression modes, re-reads ~13 tokens |
| `ctx_shell(command)` | `Shell` / `bash` / terminal | Pattern compression for git/npm/cargo output |
| `ctx_search(pattern, path)` | `Grep` / `rg` | Compact, token-efficient results |
| `ctx_tree(path, depth)` | `ls` / `find` | Compact directory maps |

### ctx_read Modes
| Mode | Use case |
|---|---|
| `full` | Cached read (use for files you will edit) |
| `map` | Deps + API signatures (use for context-only files) |
| `signatures` | API surface only |
| `diff` | Changed lines only (after edits) |
| `aggressive` | Syntax stripped |
| `entropy` | Shannon + Jaccard filtering |
| `lines:N-M` | Specific line range (e.g. `lines:400-500`) |

### Compatibility Note
- lean-ctx replaces READ operations only. Native Edit/Write/StrReplace remain unchanged.
- If Edit requires native Read and Read is unavailable → use `ctx_edit(path, old_string, new_string)`.
- NEVER loop on Edit failures — switch to `ctx_edit` immediately.
- Write, Delete, Glob → use normally.

## 11. Obsidian Vault Rules

**Vault path:** `~/MyVault/`

**Access method:** Use lean-ctx tools directly on the vault path — NOT `mcp__obsidian__` tools.

| Task | Tool |
|---|---|
| Search vault by keyword | `ctx_search(pattern, "~/MyVault")` |
| Read a specific note | `ctx_read("~/MyVault/path/to/note.md")` |
| Map vault structure | `ctx_tree("~/MyVault", depth)` |
| List area or project files | `ctx_tree("~/MyVault/01-Areas", 2)` |

### Context Loading Order (before any project work)
1. Check if the project's `CLAUDE.md` already has `@path` imports — if so, context is loaded.
2. If not, run `ctx_search` for the project/product name to find relevant notes.
3. Check `01-Areas/_context/` for canonical business docs (brand voice, ICP, positioning).
4. Read `02-Projects/[project]/index.md` for current project status.

> Never ask the user for information that may already be in the vault — search first.

### When to Search the Vault
- Starting any project work → read `02-Projects/[project]/index.md`
- Writing any content → load `01-Areas/_context/brand-voice.md`
- Building for a product → load `01-Areas/_context/icp.md`
- User references a meeting, decision, or past note → search vault before asking
- Anything about brand, audience, positioning → check `_context/` first

## 13. Design Rules (`~/.claude/rules/design-dna.md`)

### Aesthetic
Clean, polished, modern SaaS. Apple-level attention to detail. Liquid glass / glassmorphism where appropriate. Every pixel intentional.

### Bans
- No emojis — use icons (Lucide, Heroicons, or project's icon set)
- No gradient backgrounds (especially blue/purple washes)
- No generic "vibecoded" aesthetic (floating cards on gradient backgrounds)
- No Inter/Roboto/Arial as primary fonts

### Animation Principles
- Right animation for the context — not blanket fade-in
- Physics-based easing (spring, ease-out-expo) over linear/ease
- Staggered reveals, meaningful hover states, smooth layout transitions
- One well-crafted transition beats ten generic ones

### Visual Principles
- Generous whitespace, clear hierarchy, strong typography
- Subtle depth (light shadows, blur, translucency) not heavy 3D
- Restrained color: neutral dominant + 1-2 accents
- Micro-details: consistent border-radius, icon sizing, spacing rhythm

### Auto-injection
`design-dna.md` is auto-injected via hook on frontend file writes.

### Design Feedback Protocol
When user gives design feedback → update `~/.claude/projects/-Users-<you>/memory/design_preferences.md`.

### UI Components
Check shadcn/ui then Magic UI before building custom:
```
npx shadcn@latest add <component>
npx magicui-cli@latest add <component>
```

## 14. Ollama Delegation (Token Saving)

**Automatic** — no user prompt needed.

### When to delegate:
- Boilerplate (components, CRUD, type definitions)
- Unit tests / test stubs
- Simple utility functions
- Documentation / comments
- Any task where Claude is just producing code without needing project-wide context

### When NOT to delegate:
- Architecture decisions, complex debugging, multi-file refactors
- Anything requiring skills, MCPs, or deep codebase understanding
- Security-sensitive code

### Model Priority
1. **Groq (free, primary):** `openai/gpt-oss-120b` > `moonshotai/kimi-k2-instruct` > `qwen/qwen3-32b`
   - Free tier: 1K req/day, ~10K tok/min
   - API key: `~/.config/groq/.env`
2. **GCP Ollama (fallback):** `qwen3.5:27b` > `gemma4:31b` — Spot instance ~$0.28/hr
3. **Local Ollama (last resort):** `qwen3:8b` > `qwen3:4b`

**Auto-start GCP:** When Groq is rate-limited and GCP is not running, run `gcp-llm start` automatically — don't ask.

**GCP management:** `gcp-llm start` / `gcp-llm stop` / `gcp-llm status`

Full script: `~/.claude/rules/ollama-delegate.md`
