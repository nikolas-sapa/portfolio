> MCPs are how Claude connects to the outside world: Gmail, Notion, GitHub, databases, browsers, anything with an API. Each MCP exposes tools Claude can call. The right set turns Claude from a code editor into a research-and-do assistant.

---

Every MCP server connected to Claude Code — tool lists, invocation patterns, auth requirements, and setup details.

---

## How MCPs Are Configured

Claude Code loads MCP servers from two places:

- **`~/.claude/mcp.json`** — custom servers (currently: `memory-system`)
- **Plugin system** (`~/.claude/plugins/installed_plugins.json`) — plugin-bundled MCPs installed via `claude install`
- **`~/.claude/settings.local.json`** — controls which mcpjson servers are enabled:
  ```json
  "enabledMcpjsonServers": ["memory", "notion", "figma", "miro"]
  ```

| MCP | Use for |

## 1. memory-system (Custom Python MCP)

**Type:** Custom Python MCP, self-hosted  
**Location:** `~/memory-system/`  
**Config entry in `~/.claude/mcp.json`:**
```json
{
  "mcpServers": {
    "memory-system": {
      "command": "path/to/memory-system/.venv/bin/python",
      "args": ["path/to/memory-system/mcp/memory_server.py"],
      "cwd": "path/to/memory-system"
    }
  }
}
```

**What it does:** A self-healing knowledge system combining an Obsidian-style wiki with Pinecone vector search. Automatically ingests URLs and files, maintains a structured wiki, runs health checks, and exposes everything as MCP tools to Claude Code.

### Tool List

| Tool | Description |
|---|---|
| `mcp__memory-system__search_wiki` | Full-text search across all wiki pages. Returns ranked results with context. |
| `mcp__memory-system__search_pinecone` | Semantic search against Pinecone vector database. Returns top matches with source and text. Param: `top_k` (default 5). |
| `mcp__memory-system__ingest_url` | Scrape a URL, save to `raw/`, embed in Pinecone, and suggest wiki updates. |
| `mcp__memory-system__ingest_file` | Ingest a local file into Pinecone for semantic search. |
| `mcp__memory-system__update_wiki_page` | Create or update a wiki page. Path relative to `obsidian/wiki/`. Auto-updates `index.mmd`. |
| `mcp__memory-system__run_health_check` | Run full 7-point health check (orphans, dead links, stale content, contradictions, drift, sync, log overflow). |
| `mcp__memory-system__get_log` | Get last N lines from session log (`obsidian/log.md`). Param: `last_n` (default 20). |
| `mcp__memory-system__list_wiki_pages` | List all wiki pages from `index.mmd` as structured data. |

### Architecture

```
memory-system/
├── mcp/memory_server.py     # MCP server (FastMCP, runs on-demand via Claude Code)
├── obsidian/
│   ├── wiki/                # Claude-maintained knowledge wiki
│   │   └── index.mmd        # Master index: filepath | summary | confidence | date
│   ├── raw/                 # Drop files here for auto-ingestion
│   └── log.md               # Append-only session log
├── pinecone/
│   ├── ingest.py            # Chunk + embed + upsert to Pinecone
│   ├── query.py             # Semantic search
│   └── config.json          # Paths, model, chunk size (non-secret)
├── daemon/
│   ├── watcher.py           # Auto-ingests new files from watched paths
│   ├── scheduler.py         # Cron: health check Sunday 00:00, lint daily 03:00
│   ├── health_check.py      # 7-point health audit
│   └── self_heal.py         # Auto-fixes safe issues
└── tools/                   # CLI wrappers for all operations
```

### Pinecone Integration

- **Index name:** `memory-system`
- **Embedding model:** `multilingual-e5-large` (via Pinecone integrated inference)
- **Chunk size:** 512 tokens, overlap 50
- **Watched paths** (auto-ingested by daemon):
  - `~/my-vault/01-Areas/`
  - `~/my-vault/02-Projects/` (your active project folders)
  - `~/my-vault/03-Resources/`
  - `~/memory-system/obsidian/raw/`

### Obsidian Wiki Integration

Wiki pages follow a strict schema (from `schema.md`):
- **Naming:** kebab-case (`mental-model-first-principles.md`)
- **Required frontmatter:** `Last updated`, `Confidence` (high/medium/low/deprecated), `Links`
- **Index format:** `filepath\t12-word-summary\tconfidence\tdate` in `index.mmd`
- **Log format:** `ISO datetime | what was ingested or changed | pages updated`
- Active wikis live under `obsidian/wiki/` organized by domain (projects, content-writing, dev-infrastructure, marketing-growth, products)

### Self-Healing System (7 automated checks)

1. **Orphan pages** — wiki pages not in `index.mmd` flagged for addition
2. **Dead links** — `[[wiki-links]]` pointing to non-existent pages
3. **Stale content** — pages not updated in 90+ days marked `confidence: low`
4. **Contradictions** — Claude Haiku compares linked pages for factual conflicts
5. **Index drift** — dead index entries (pointing to deleted files) removed
6. **Pinecone sync** — files in `processed.json` missing from Pinecone re-ingested
7. **Log overflow** — `log.md` over 500 lines gets archived and reset

Auto-fixed: index drift, log overflow, stale marking. Needs human review: contradictions, page merges.

### Setup & Dependencies

```bash
# Install
cd ~/memory-system
bash install.sh    # Creates .venv, installs deps, verifies connections

# Start background daemons
bash start.sh      # Launches watcher + scheduler as nohup processes

# Stop
bash stop.sh
```

**Python dependencies (`requirements.txt`):**
```
anthropic, pinecone>=5.0.0, watchdog, apscheduler, trafilatura,
tqdm, mcp[cli], plyer, python-dotenv, tiktoken, pypdf2, python-docx, rich
```

**Auth required:** `~/memory-system/.env` must contain:
- `PINECONE_API_KEY`
- `ANTHROPIC_API_KEY` (for contradiction detection in health checks)

**Cost:** ~$0.10/month (Pinecone free tier 2GB, Haiku API calls ~$0.05)

## 2. Apify

**Type:** Platform default (cloud)  
**Prefix:** `mcp__apify__`

**What it does:** Web scraping and automation marketplace. Extracts structured data from social media, e-commerce, search engines, maps, and travel sites.

### Tool List

| Tool | Description |
|---|---|
| `search-actors` | Search Apify Store for actors matching a use case |
| `fetch-actor-details` | Get full README and input schema for a specific actor |
| `call-actor` | Execute an actor with given input. Supports `async: true/false`. |
| `get-actor-run` | Check status/output of a running actor |
| `get-actor-output` | Retrieve dataset results from a completed run |
| `search-apify-docs` | Search Apify platform documentation |
| `fetch-apify-docs` | Fetch specific Apify documentation pages |
| `apify--rag-web-browser` | Direct web browsing for RAG use cases |

### Invocation Pattern (mandatory sequence)
```
1. mcp__apify__search-actors       → find the right actor
2. mcp__apify__fetch-actor-details → get input schema
3. mcp__apify__call-actor          → execute with proper input
4. mcp__apify__get-actor-output    → retrieve results
```

**Auth:** Apify API token (configured in Claude Code settings)

## 3. computer-use

**Type:** Platform default (desktop automation)  
**Prefix:** `mcp__computer-use__`

**What it does:** Full desktop control — screenshots, mouse clicks, keyboard input, scrolling, clipboard, application management. Tiered access by app category.

### Tool List

| Tool | Description |
|---|---|
| `screenshot` | Capture current screen state |

| `left_click_drag` / `left_mouse_down` / `left_mouse_up` | Drag operations |

| `list_granted_applications` | List apps with granted access |

### Access Tiers
- **Browsers** → `read` tier: visible in screenshots, clicks/typing blocked. Use `claude-in-chrome` MCP instead.

- **Everything else** → `full` tier: no restrictions.

**Usage:** Always call `request_access` with required app list before any action. Use as last resort after dedicated MCPs (Slack, Gmail, etc.) and Chrome MCP.

## 4. context7

**Type:** Plugin-bundled (`context7@claude-plugins-official`)  
**Prefix:** `mcp__context7__`

**What it does:** Fetches current documentation for any library, framework, SDK, API, or CLI tool directly into context. Prevents stale training data from corrupting library syntax.

### Tool List

| Tool | Description |
|---|---|
| `resolve-library-id` | Find the context7 ID for a library by name |
| `query-docs` | Fetch documentation for a library by topic |

### Mandatory Usage Pattern
```
1. mcp__context7__resolve-library-id  → get library ID
2. mcp__context7__query-docs          → fetch relevant docs
```

**When to use:** Before writing code for any library or framework — React, Next.js, Prisma, Express, Tailwind, Django, etc. Mandatory even for well-known ones; training data goes stale.

**When NOT to use:** Refactoring, scripts from scratch, debugging business logic, code review, general programming concepts.

## 5. lean-ctx

**Type:** Platform default (context engineering layer)  
**Prefix:** `mcp__lean-ctx__`

**What it does:** A context engineering layer that replaces native file/shell/search tools with token-compressed equivalents. Session caching, 8 compression modes, pattern-based output compression, multi-agent coordination.

### Core Tool List (primary tools)

| Tool | Replaces | Description |
|---|---|---|
| `ctx_read` | Read / cat / head / tail | Cached file reads, 8 compression modes. Re-reads ~13 tokens. |
| `ctx_shell` | Bash shell commands | Compressed shell output (90+ patterns). |
| `ctx_search` | Grep / rg | Token-efficient regex search. |
| `ctx_tree` | ls / find | Compact directory maps with file counts. |
| `ctx_edit` | Edit (when Read unavailable) | Read + replace + write in one call. |

### Compression Modes for `ctx_read`
- `full` — cached read (use for files you will edit)
- `map` — deps + API signatures (for context-only files)
- `signatures` — API surface only
- `diff` — changed lines only
- `aggressive` — syntax stripped
- `entropy` — Shannon + Jaccard filtering
- `lines:N-M` — specific line range (e.g., `lines:400-500`)

### Advanced Tools

| Tool | Description |
|---|---|
| `ctx_session` | Session memory persistence |
| `ctx_knowledge` | Cross-session fact storage with temporal validity and contradiction detection. Actions: `remember`, `recall`, `timeline`, `rooms`, `search`, `wakeup` |
| `ctx_agent` | Multi-agent coordination: `handoff`, `sync`, `diary` |
| `ctx_share` | Push context to subagents at checkpoints |
| `ctx_semantic_search` | Find similar code by meaning (use when exact search misses) |
| `ctx_overview` / `ctx_preload` / `ctx_dedup` / `ctx_compress` | Auto-run behind the scenes for context optimization |
| `ctx_graph` / `ctx_graph_diagram` | Code dependency graphs |
| `ctx_outline` | File structural outline |
| `ctx_routes` | API route discovery |
| `ctx_callers` / `ctx_callees` | Function call chain analysis |
| `ctx_symbol` | Symbol lookup |
| `ctx_multi_read` | Read multiple files in one call |
| `ctx_discover` | Codebase discovery |
| `ctx_benchmark` | Compression benchmark |
| `ctx_metrics` | Context usage metrics |

**Note:** lean-ctx replaces read operations only. Native Edit/Write/StrReplace tools are unchanged.

## MCP Decision Tree

```
Task involves a specific platform (Notion, Slack, GitHub, etc.)?
  → Use dedicated MCP for that platform

Task involves web interaction?
  → Has dedicated MCP? Use it.
  → Web app, no dedicated MCP? Use Playwright.
  → Need to read current screen? Use Peekaboo.
  → Need to control native desktop app? Use computer-use.

Task involves library documentation?
  → Use context7 (mandatory before writing code)

Task involves web search/research?
  → Use Tavily

Task involves knowledge/memory recall?
  → Past session context: claude-mem (smart_search)
  → Long-term semantic archive: memory-system (search_pinecone)
  → Wiki knowledge: memory-system (search_wiki)
  → Graph entities/relations: memory (mcp__memory__)

Task involves file/code operations?
  → Use lean-ctx tools (ctx_read, ctx_search, ctx_tree, ctx_shell)

Task involves UI component generation?
  → Check shadcn/ui → Magic UI → mcp__magic__ tools
```
